# RISK & DRIFT WARNINGS

## 1. 文档目的

本文档用于定义 TraceFold 在 **Phase 1** 阶段的主要风险、常见偏航路径、红线行为与纠偏原则。

它不是功能规划文档，也不是架构细节说明文档，而是一个**防跑偏约束文档**。

当实现选择、目录演化、任务拆分或技术决策开始偏离既定目标时，应优先参考本文档进行判断。

---

## 2. 适用范围

本文档适用于 TraceFold 的 Phase 1 基础平台建设阶段，适用于：

- 项目结构设计
- API 工程搭建
- 公共层建设
- 业务域接入顺序与方式
- Codex 任务拆分与执行
- 配置、数据、目录与依赖关系判断

---

## 3. 当前阶段的唯一核心目标

Phase 1 的目标不是尽快做出“大量功能”，而是先建立一个可维护、可扩展、可持续交给 Codex 实现的稳定底座。

当前阶段唯一正确的主线是：

1. 固定文档基线
2. 固定架构规则
3. 建立统一 API 骨架
4. 建立公共层
5. 建立 SQLite 统一真相源
6. 验证 1~3 个业务域可以共用同一套底座

任何偏离这条主线、并试图以“更快看到效果”为理由提前扩展复杂能力的行为，都应视为偏航风险。

---

## 4. 风险判断总原则

如果某项改动满足以下任一特征，应视为高风险：

- 让局部实现便利压过整体结构稳定
- 让短期演示效果压过长期可维护性
- 让客户端形态反向决定底层架构
- 让 AI / 自动化能力反向定义正式数据结构
- 在没有文档更新的情况下先改代码方向
- 为未来可能性提前引入当前阶段不需要的复杂度

---

## 5. 主要偏航风险

### 5.1 功能堆叠型偏航

表现：

- 还没建立稳定公共层，就开始大面积写业务功能
- 还没验证基础 CRUD 闭环，就想同时推进多个业务域
- 为了“先能用”而绕过统一结构直接落代码

风险：

- 代码越来越多，但底层越来越乱
- 新增业务域时不断复制已有代码
- 后续重构成本快速上升

纠偏原则：

- 先把公共层立住，再扩业务
- 先验证统一模式，再增加业务域数量
- 优先减少结构漂移，不追求短期功能数量

---

### 5.2 客户端反向绑架底层型偏航

表现：

- 因为想尽快看到 UI，就先做 Web / Desktop 壳
- 因为客户端交互方便，而临时改动 API 结构
- 让桌面端、前端、Bot 的接入方式反向决定后端底座

风险：

- API 失去统一性
- 不同客户端推动不同底层实现
- 底座变成被界面牵着走的拼接工程

纠偏原则：

- Phase 1 必须坚持 API 先行
- 客户端只能消费边界清晰的 API
- 不允许先做壳，再倒逼底层结构重写

---

### 5.3 公共能力复制扩散型偏航

表现：

- 各业务域自行处理日志、异常、响应格式、数据库会话
- 业务域中重复出现“差不多”的底座代码
- 为了少写公共封装，直接在各域里复制实现

风险：

- 统一性丧失
- 后期维护成本剧增
- Codex 后续输出风格越来越不一致

纠偏原则：

- 共性能力必须公共化
- 不允许业务域私自复制底座能力
- 发现第二次相似实现时，应优先考虑公共层收口

---

### 5.4 过度设计型偏航

表现：

- 提前引入多租户、复杂权限、消息队列、缓存、微服务、插件系统
- 为“未来可能会有”而设计大量抽象层
- 在单机、本地优先阶段引入分布式思维

风险：

- 复杂度远超当前收益
- 骨架迟迟落不了地
- Phase 1 被长期拖延

纠偏原则：

- 当前阶段默认拒绝高级复杂度
- 没有被当前主线直接需要的复杂机制，一律后移
- 先把可运行底座做实，再考虑升级方向

---

### 5.5 数据真相源漂移型偏航

表现：

- 某业务域单独引入其他数据库或特殊数据源
- 把缓存、中间文件、临时导出文件当成正式记录来源
- 出现多个“实际都在改数据”的地方

风险：

- 数据一致性失控
- 追踪问题来源变难
- 后续自动化和多端接入会建立在不稳定事实层上

纠偏原则：

- Phase 1 的结构化数据统一进入 SQLite
- SQLite 是唯一真相源
- 缓存、中间文件、导入导出结果都不是正式事实层

---

### 5.6 文档失效型偏航

表现：

- 规则文档和代码实现长期不一致
- 先改代码，文档以后再补
- 任务描述越来越依赖聊天上下文，而不是仓库文档

风险：

- 项目开始失去统一基线
- Codex 无法稳定接手后续任务
- 新任务越来越依赖口头补丁说明

纠偏原则：

- 重大结构变化必须先更新文档
- 新增配置必须同步更新 env 规范
- 新增业务域必须先确认是否符合已有规则文档

---

### 5.7 AI 能力抢跑型偏航

表现：

- 在底座未稳时优先做 AI 分析、自动总结、智能派生
- 让 AI 输出直接充当正式事实记录
- 为了 AI 方便而强行改动数据结构

风险：

- 正式数据层与派生层混淆
- 系统结构被不稳定能力反向塑形
- 可解释性与可维护性下降

纠偏原则：

- AI 能力只能建立在稳定底座之上
- AI 派生内容不能替代正式记录
- 任何 AI 接入都不得反向破坏事实层边界

---

## 6. 红线行为（出现即判定为明显偏航）

以下行为在 Phase 1 中视为明确红线：

1. 在未完成基础骨架前，直接堆叠大量业务功能
2. 先做 Web / Desktop / Bot 主体，再倒推 API 设计
3. 在业务域内复制日志、异常、数据库底座等公共能力
4. 在 SQLite 之外引入新的正式真相源
5. 为未来假设提前引入微服务、消息队列、缓存、插件化等复杂系统
6. 在未更新文档的情况下直接进行重大结构改动
7. 让 AI 输出直接成为正式业务数据
8. 为局部实现方便而改变已定项目方向
9. 让目录结构随着任务推进不断漂移
10. 用一次性特判替代统一模式

---

## 7. 可观测的偏航信号

如果仓库开始出现以下现象，应视为已经接近偏航或正在偏航：

- 相似的共性代码在多个业务域重复出现
- 新增功能越来越依赖临时说明才能接上
- 目录命名和职责解释开始不一致
- 某些功能能跑，但无法解释其在整体结构中的位置
- 新增一个业务域需要改很多底层公共代码
- 配置项越来越多，但没有统一文档入口
- 业务推进速度看似很快，但基础层仍然不稳定
- 代码比文档增长快很多，且文档长期滞后

---

## 8. 出现偏航时的处理顺序

一旦发现偏航迹象，应按以下顺序处理：

1. 先暂停继续堆功能
2. 明确偏航发生在哪一层：文档、结构、数据、配置、业务域、客户端边界
3. 回到 `PROJECT_BRIEF.md` 检查当前阶段目标是否被改变
4. 回到 `ARCHITECTURE_RULES.md` 检查是否触碰结构红线
5. 必要时先修正文档，再修正代码
6. 只有在结构重新稳定后，才恢复后续实现任务

---

## 9. 每次任务开始前的自检问题

在执行任何新任务前，至少先回答以下问题：

1. 这个任务是在加强底座，还是在偷跑功能？
2. 这个改动会不会让客户端反向绑架底层？
3. 这个实现有没有复制本应属于公共层的能力？
4. 这个改动是否引入了当前阶段不需要的复杂度？
5. 这个改动是否改变了正式数据层边界？
6. 文档是否需要先更新？
7. 这个任务完成后，系统会更稳定，还是只会“看起来更完整”？

如果其中任一问题无法明确回答，说明任务定义仍不够稳，不应直接进入实现。

## 10. 与其他文档的关系

- 项目定位、阶段目标与阶段边界，以 `PROJECT_BRIEF.md` 为准
- 架构实现约束与目录/依赖红线，以 `ARCHITECTURE_RULES.md` 为准
- 数据实体与字段语义，以 `DATA_MODEL_DRAFT.md` 为准
- Phase 1 实施顺序与执行路线，以 `ROADMAP_PHASE1.md` / `阶段一规划` 为准
- 任务拆分与交付格式，以 `TASK_TEMPLATE.md` 为准
- 环境变量与配置约束，以 `ENV_CONVENTIONS.md` 为准

当实际代码与上述文档发生冲突时，应优先更新文档基线，再决定是否修改代码。

---

## 11. 本文档的定位

本文档是 TraceFold Phase 1 的正式约束文档之一。

它的作用不是替代项目说明、架构规则或路线图，而是专门负责回答以下问题：

- 当前阶段最容易怎么跑偏
- 哪些行为属于高风险实现方式
- 哪些行为已经触碰红线
- 发现偏航后应如何纠正

Phase 1 期间，任何重大结构改动、任务拆分与技术取舍，都应同时接受本文档的风险判断。
