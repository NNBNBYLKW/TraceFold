# PROJECT BRIEF

## 1. 项目名称

TraceFold

---

## 2. 项目定位

TraceFold 是一个**本地优先（local-first）**的个人数据平台基础版项目。

它的第一阶段目标，不是做一个功能很多、界面很多、系统很大的“大而全产品”，而是先建立一个**可长期扩展的稳定底座**，让后续可以在不推翻架构的前提下，逐步接入更多业务能力与客户端。

TraceFold 在当前阶段的核心价值是：

- 为个人数据提供统一落点
- 为不同业务域提供一致的数据与 API 底座
- 为未来 Web / Desktop / Bot 接入预留清晰边界
- 为后续自动化、规则能力、AI 辅助能力提供可扩展基础

---

## 3. 当前阶段判断

TraceFold 当前处于 **Phase 1：基础平台建设阶段**。

这一阶段的重点不是“功能堆叠”，而是：

- 先把 API 骨架搭稳
- 先把公共层立好
- 先把 SQLite 数据底座跑通
- 先验证 1~3 个业务域能稳定共用一套底座

换句话说，Phase 1 的目标是：

**先建立可维护的系统结构，再逐步增加业务能力。**

---

## 4. 第一阶段将覆盖的业务方向

Phase 1 先围绕以下业务域建立基础能力：

- `capture`：快速记录
- `pending`：待处理事项
- `expense`：消费记录
- `knowledge`：知识条目 / 笔记索引
- `health`：健康记录

这些业务域当前并不是都要一次性完整做完，而是作为统一架构下的预留方向。

建议的实际落地优先顺序为：

1. `capture`
2. `pending`
3. `expense`
4. `knowledge`
5. `health`

---

## 5. 第一阶段核心目标

Phase 1 只关注以下核心目标：

1. **建立清晰稳定的 API 工程结构**
2. **建立统一公共层**
3. **使用 SQLite 作为唯一真相源**
4. **建立统一配置、日志、异常、响应、数据库会话机制**
5. **按业务域组织代码，但共性能力必须公共化**
6. **验证至少 1~3 个业务域可共用同一套底座**
7. **为后续 Web / Desktop / Bot 接入预留清晰边界**
8. **让后续代码任务可以稳定交给 Codex 执行**

---

## 6. 第一阶段非目标

以下内容不属于 Phase 1 的重点：

- 不追求完整权限系统
- 不追求复杂认证体系
- 不追求多租户设计
- 不追求分布式部署
- 不追求消息队列、缓存、微服务
- 不追求高级插件系统
- 不追求一次性把所有业务规则做完
- 不追求完整 AI 管线
- 不追求复杂搜索系统
- 不追求为了“未来可能会用到”而过度设计数据库

Phase 1 的原则很明确：

**先把底座做稳，不抢跑高级能力。**

---

## 7. 核心产品立场

### 7.1 本地优先

TraceFold 的第一阶段是本地优先系统：

- 优先服务本地运行
- 优先服务个人工作流
- 优先服务单机开发与验证
- 不为远程分布式场景提前引入复杂度

### 7.2 单用户优先

当前唯一目标用户是项目作者本人。

这意味着第一阶段可以优先服务：

- 个人记录
- 个人管理
- 个人整理
- 个人自动化流程

而不需要为团队协作、多账号、组织权限提前增加复杂性。

### 7.3 API 先行

第一阶段优先建立 API 底座，而不是先做完整前端。

原因：

- API 是后续各客户端的统一连接层
- API 先稳定，后续 Web / Desktop / Bot 才不会各自长出一套底层
- 更适合通过规则文档 + 任务模板约束 Codex 持续实现

## 8. 技术方向（Phase 1）

### API

- Python
- FastAPI

### Database

- SQLite
- SQLAlchemy

### Config

- `.env`
- 统一配置入口：`app/core/config.py`

### Architecture

- API 采用“**业务域 + 公共层**”并存结构
- 业务域代码放在：`app/domains/*`
- 公共基础设施放在：
  - `app/core`
  - `app/db`
  - `app/schemas`
  - `app/api`

---

## 9. 架构原则

### 原则 1：业务按域拆分

每个业务域独立维护自己的：

- `router.py`
- `service.py`
- `repository.py`
- `models.py`
- `schemas.py`

### 原则 2：底座按公共层收口

以下能力必须统一放在公共层：

- config
- logging
- exceptions
- common responses
- db base / session / init
- shared common schemas
- API 总路由与系统级接口

### 原则 3：SQLite 是唯一真相源

第一阶段所有结构化数据统一进入 SQLite。

不允许：

- 各模块自行引入额外数据库
- 各业务域维护自己的独立真相源
- 把缓存或中间文件当成正式数据源

### 原则 4：先保证可维护，再追求功能速度

TraceFold 第一阶段优先保证：

- 目录清晰
- 依赖方向稳定
- 规则可执行
- 新增业务域不需要重构底层

### 原则 5：事务由服务层控制

默认事务边界由 `service` 控制，而不是由 `router` 或 `repository` 零散处理。

这样可以保证后续：

- 一个用例协调多个 repository
- 统一提交 / 回滚逻辑
- 更稳定的扩展与测试

## 10. 目标用户与使用场景

当前唯一目标用户是项目作者本人。

当前主要场景包括：

- 快速记录临时信息
- 管理待处理事项
- 记录消费与支出
- 整理知识条目
- 记录健康数据
- 为后续自动化工具、脚本、Bot 提供统一 API 底座

这决定了第一阶段的产品策略：

- 优先实用
- 优先可维护
- 优先可扩展
- 不追求过度通用化

---

## 11. 当前开发方式

TraceFold 在第一阶段的一个重要前提是：

**后续大量代码实现将交由 Codex 执行。**

因此，这个项目不能只靠“临时口头说明”推进，而必须靠文档约束实现质量。

当前仓库必须具备以下约束能力：

- 清晰目录结构
- 明确架构规则
- 稳定的数据模型草案
- 固定任务模板
- 统一 env 规范
- 清晰的阶段路线图

换句话说：

**文档本身就是底座的一部分。**

---

## 12. 对 Codex 的要求

TraceFold 的实现任务必须尽量做到：

- 目标明确
- 修改范围明确
- 架构约束明确
- 配置变更规则明确
- 交付清单明确
- 验收标准明确

Codex 不是自由发挥式开发，而是：

**在清晰规则下完成实现。**

因此本项目默认要求：

1. 先写规则文档，再让 Codex 写代码
2. 新增配置必须同步文档
3. 新增业务域必须遵循统一结构
4. 不允许绕过公共层直接复制底座能力
5. 重大结构变化必须先修改架构文档

---

## 13. 第一阶段的最小成功标准

如果 Phase 1 结束时满足以下条件，则可视为基础平台搭建成功：

- API 项目结构清晰且稳定
- 公共层已成型
- 所有配置通过统一入口读取
- 所有业务域复用统一 DB base / session
- SQLite 作为统一真相源工作正常
- 能实现至少 1~3 个业务域的基础 CRUD
- 新增一个业务域不需要重构公共底座
- 后续 Web / Desktop / Bot 能通过 API 清晰接入
- Codex 可以在既定规则下持续实现后续任务而不频繁跑偏

---

## 14. 当前阶段的边界判断

在当前阶段，TraceFold 更像是：

- 一个**个人数据平台底座**
- 一个**统一 API 基础层**
- 一个**后续多端接入的中枢**
- 一个**适合持续迭代的个人系统工程**

而不是：

- 一个完整成熟的商业 SaaS
- 一个多团队协作系统
- 一个复杂企业级平台
- 一个已经完成所有产品定义的最终形态产品

---

## 15. 当前最重要的事情

在当前阶段，最重要的不是尽快写大量业务代码，而是先把下面这些东西立住：

1. 项目说明文档
2. 架构规则文档
3. 任务模板
4. 数据模型草案
5. 阶段路线图
6. env 规范
7. API 公共层骨架

只有这些东西立住后，后续让 Codex 持续接手实现，才不会越来越乱。
