# MIGRATION MIN SPEC

## 1. 目标

本文档定义 TraceFold 在 Phase 1 / Step 2 阶段的**最小数据库迁移规范**。

当前目标不是立刻引入重型迁移平台，而是先明确：

- 表如何进入系统
- 表变更如何记录
- 初始化与变更边界如何区分
- 哪些修改允许直接通过 `init_db.py` 承载
- 什么情况下必须升级到正式迁移工具

本规范服务于当前阶段的底座收口，防止数据库结构“先改再说”。

---

## 2. 当前适用范围

当前规范适用于：

- `apps/api/app/db/base.py`
- `apps/api/app/db/init_db.py`
- `apps/api/app/domains/*/models.py`
- `docs/DATA_MODEL_DRAFT.md`

当前正式真相源关系如下：

1. **运行真相源**：SQLite 数据库文件
2. **结构声明源**：SQLAlchemy 模型定义
3. **初始化入口**：`app.db.init_db:init_db()`
4. **结构变更记录源**：`docs/DATA_MODEL_DRAFT.md`

当前阶段尚**未正式引入 Alembic**。因此，数据库结构变更不能被当作“自由修改代码后的自然结果”。

---

## 3. 当前阶段的最小迁移策略

### 3.1 初始化与迁移的区分

当前必须明确区分两件事：

- **初始化（init）**：在空数据库上创建当前已实现的表
- **迁移（migration）**：在已有数据库上对表结构做变更

`init_db.py` 只承担：

- 注册当前已真实存在的模型
- 在空库上创建表
- 为本地开发提供最小初始化入口

`init_db.py` **不应被假装当作完整迁移系统**。

---

### 3.2 当前阶段允许的变更方式

在 Phase 1 / Step 2 当前状态下，允许的数据库结构变更方式为：

#### 允许

1. 修改 SQLAlchemy 模型定义
2. 同步更新 `docs/DATA_MODEL_DRAFT.md`
3. 在开发环境中**重建本地 SQLite 数据库**
4. 重新运行：

```powershell
python -c "from app.db.init_db import init_db; init_db()"
```

也就是说，当前阶段允许的正式策略是：

**“开发期 schema 变更 + 文档同步 + 重建本地库”**

这是一种明确受限、可接受的 **Phase 1 最小迁移规范**，不是长期正式方案。

---

### 3.3 当前阶段不允许的做法

以下行为当前不允许：

1. 直接修改数据库文件结构，但不改模型
2. 改了模型，但不更新 `DATA_MODEL_DRAFT.md`
3. 改了表结构，但不说明旧数据如何处理
4. 让业务代码自己临时执行 DDL
5. 把“先删库重建”当成长期正式迁移机制
6. 在多个地方各自维护自己的建表逻辑

---

## 4. 结构变更的最小执行流程

当需要新增表、改字段、改索引时，当前阶段按下面流程执行：

### Step A：先改模型

在对应的 `models.py` 中完成结构定义修改。

### Step B：同步文档

更新 `docs/DATA_MODEL_DRAFT.md`，至少补充：

- 表名
- 字段
- 主键
- 约束
- 本次变更说明
- 如果有，说明与旧字段的替代关系

### Step C：检查 `init_db.py`

确认新模型已被当前正式初始化入口导入。

要求：

- 只导入当前已真实存在、已准备启用的模型
- 不为“未来可能会做”的业务域提前接入空模型

### Step D：重建本地数据库并验证

开发环境下删除旧的本地 SQLite 数据库文件或使用新的测试库路径，然后执行：

```powershell
python -c "from app.db.init_db import init_db; init_db()"
```

### Step E：最小验证

至少验证：

1. 表能创建成功
2. 服务能正常启动
3. 最小系统测试仍能通过
4. 如涉及任务表或业务表，最小链路可读写

---

## 5. 与测试和验证的关系

任何涉及数据库结构的改动，至少需要重新执行：

```powershell
python -m pytest
```

如果当前仅调整系统底座，也至少应通过：

```powershell
python -m pytest tests/test_system.py
```

如果某个业务域已有 smoke test，则相关变更后也必须重新跑对应测试。

---

## 6. 何时必须升级到正式迁移工具

当出现以下任一情况时，当前最小迁移规范不再够用，应进入下一阶段并接入正式迁移工具（优先考虑 Alembic）：

1. 需要保留已有开发数据，不能再依赖删库重建
2. 同一表开始频繁增删改字段
3. 出现字段重命名、数据搬迁、默认值回填等操作
4. 多个业务域并行推进，结构变更频率明显上升
5. 需要可追踪的迁移历史，而不只是文档说明

也就是说：

**当前规范是 Step 2 的收口方案，不是 Phase 1 全程的最终迁移方案。**

---

## 7. 当前阶段验收口径

在当前 Step 2 验收下，“迁移机制或最小迁移规范已建立” 的判定标准为：

- [ ] 已存在明确的迁移规范文档
- [ ] `init_db.py` 只负责初始化，不被误认为正式迁移系统
- [ ] 模型变更必须同步 `DATA_MODEL_DRAFT.md`
- [ ] 当前开发期允许通过重建本地 SQLite 完成 schema 更新
- [ ] 已明确写出何时必须升级到正式迁移工具

---

## 8. 当前结论

TraceFold 在 Step 2 当前阶段采用的数据库变更策略为：

**最小迁移规范 + 文档同步 + 开发环境重建本地库验证**

它满足当前 Phase 1 对“数据库结构变化不能无规则发生”的要求，
同时避免了在底座尚薄时过早接入重型迁移体系。
